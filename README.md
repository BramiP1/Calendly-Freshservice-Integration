# Calendly → FreshService Connector

A lightweight Node.js/Express webhook server that listens for Calendly booking events and automatically creates or updates tickets in FreshService.

---

## Scope

| Calendly Event       | FreshService Action                                              |
|----------------------|------------------------------------------------------------------|
| `invitee.created`    | Create a new ticket with meeting details and invitee info        |
| `invitee.canceled`   | Add a cancellation note + close the ticket (or create one if not found) |

## Setup

### 1. Install dependencies

```bash
npm install
```

### 2. Configure environment variables

```bash
cp .env.example .env
```

Edit `.env` and fill in all values:

| Variable                          | Description                                              |
|-----------------------------------|----------------------------------------------------------|
| `PORT`                            | Local server port (default: 3000)                        |
| `CALENDLY_WEBHOOK_SIGNING_KEY`    | Signing key from your Calendly webhook subscription      |
| `CALENDLY_API_TOKEN`              | Calendly Personal Access Token                           |
| `FRESHSERVICE_DOMAIN`             | e.g. `yourcompany.freshservice.com`                      |
| `FRESHSERVICE_API_KEY`            | FreshService API key (found in Profile Settings)         |
| `FRESHSERVICE_SERVICE_ITEM_ID`    | Numeric ID of the Service Catalog item (see below)       |
| `FRESHSERVICE_DEFAULT_PRIORITY`   | Fallback priority if not using a service item: 1=Low, 2=Medium, 3=High, 4=Urgent |
| `FRESHSERVICE_DEFAULT_EMAIL`      | Fallback requester email if invitee email is missing     |

**Where to find your keys:**
- **Calendly API Token**: Calendly → Integrations → API & Webhooks → Personal Access Token
- **Calendly Signing Key**: Set when creating a webhook subscription (auto-generated by Calendly)
- **FreshService API Key**: FreshService → Profile Settings → API Key (bottom of page)
- **FreshService Service Item ID**: see section below

## Setting Up the FreshService Service Item

This connector files Calendly bookings as **Service Requests** under a specific item in your FreshService Service Catalog. Here is how to configure it:

### Step 1 — Create (or find) a Service Item in FreshService

1. In FreshService, go to **Admin → Service Catalog**
2. Either create a new Service Item (e.g. "Calendly Meeting Request") or pick an existing one
3. Open the item to edit it — look at the **URL in your browser**:
   ```
   https://yourcompany.freshservice.com/service_catalog/items/42/edit
   ```
   The number in the URL (`42` in this example) is your **Service Item ID**

### Step 2 — Set the ID in your environment

In your `.env` (locally) or in Render's Environment settings (in production):

```
FRESHSERVICE_SERVICE_ITEM_ID=42
```

### What happens at runtime

- **`invitee.created`** → A Service Request is created under that Service Item, with the requester set to the invitee's email. Meeting details (event name, time, join link, cancel/reschedule URLs) are posted immediately as a note on the request so agents see them in the conversation thread.
- **`invitee.canceled`** → The original Service Request is found by its unique Calendly event tag, a cancellation note is added, and the request is closed.

> You do **not** need custom fields on the Service Item. All meeting data is delivered via the note. Routing to a specific group or agent can be configured directly on the Service Item in FreshService Admin — the connector does not need to know about it.

## Running Locally with ngrok

### Step 1 — Start the server

```bash
npm run dev
```

The server starts at `http://localhost:3000`.

### Step 2 — Start ngrok

In a separate terminal:

```bash
ngrok http 3000
```

Copy the `Forwarding` URL — e.g. `https://abc123.ngrok-free.app`.

### Step 3 — Register the Calendly webhook

```bash
node src/scripts/registerWebhook.js https://abc123.ngrok-free.app
```

This registers `invitee.created` and `invitee.canceled` events pointing at your ngrok URL. The signing key from your `.env` is used automatically.

### Step 4 — Test it

Book or cancel a meeting on your Calendly page and watch the server logs. A ticket will appear in FreshService within seconds.

## Deploying to Render (24/7)

### Step 1 — Push the project to GitHub

Create a new GitHub repo and push this folder to it. Render deploys directly from a branch.

### Step 2 — Create a Render Web Service

1. Go to [render.com](https://render.com) → **New → Web Service**
2. Connect your GitHub repo
3. Set the following in Render's build/start settings:

| Setting        | Value           |
|----------------|-----------------|
| **Environment** | `Node`         |
| **Build Command** | `npm install` |
| **Start Command** | `npm start`   |

### Step 3 — Set environment variables in Render

In the Render dashboard → your service → **Environment**, add every key from `.env.example` with your real values. Do **not** commit your `.env` file.

### Step 4 — Keep it running 24/7

**Free tier caveat:** Render's free tier spins down web services after ~15 minutes of no traffic. When it spins back up there's a cold-start delay of 30–60 seconds — long enough for Calendly to consider the webhook delivery failed and retry.

**Fix (free tier) — UptimeRobot ping:**
1. Sign up at [uptimerobot.com](https://uptimerobot.com) (free)
2. Add a new **HTTP(s) monitor**:
   - URL: `https://your-service.onrender.com/health`
   - Interval: **every 5 minutes**
3. This keeps the server alive by hitting the `/health` endpoint before Render's idle timeout triggers

**Fix (paid tier) — upgrade to Render Starter ($7/month):**
- Services on paid plans never spin down — no ping needed
- Recommended if this connector handles any real volume

### Step 5 — Register the Calendly webhook pointing at Render

Once the service is live, run this **once** from your local machine:

```bash
node src/scripts/registerWebhook.js https://your-service.onrender.com
```

This replaces any ngrok webhook subscription with your permanent Render URL.

## How Calendly Events Are Matched to the Correct FreshService Ticket

Each Calendly scheduled event has a globally unique URI, e.g.:
```
https://api.calendly.com/scheduled_events/ABCDEF123456
```

When `invitee.created` fires, the connector extracts the UUID from that URI and adds it as a tag on the FreshService ticket:
```
calendly-evt-ABCDEF123456
```

When `invitee.canceled` fires for the same event, the connector searches FreshService for the ticket with that exact tag — not by name or email — so there is always a 1:1 match regardless of how many meetings the same person has booked.

**Why this matters:** if you search by email alone, someone who books 3 meetings could have the wrong ticket closed on a cancellation. The UUID tag makes it exact.

## Security Notes

- Webhook payloads are verified using HMAC-SHA256 against `CALENDLY_WEBHOOK_SIGNING_KEY`. Requests with invalid or missing signatures are rejected with `401`.
- Replay attacks are blocked — webhooks older than 5 minutes are rejected.
- API keys are loaded exclusively from `.env` and never hardcoded.
- `.env` is git-ignored.
